# NJM-70015 — [FLB v1] FLB - Functional - Verify User Story: Recover Files/Folders from Backup (US3)

- **Source TC:** NJM-70015 (Xray, FLB/Share, Blocker P1, flb-v1)
- **Feature:** File-Level Backup (physical machine)
- **Appliance:** nbr-84 (FLB) — see `test-data/environment.md`
- **Metadata:** testcase_id=NJM-70015, author=tri.ton, date=2026-07-08, product=FLB, status=GENERATED
- **Feasibility:** ⚠️ PARTIAL — the Xray scenario assumes source files were *accidentally deleted*
  and then restored. `TestData_ForFLB` is a **shared fixture reused by every case** in this repo —
  deliberately deleting real files from it to simulate "accidental deletion" would corrupt that
  shared fixture for other runbooks. Substitute assertion: verify the files are recoverable and
  byte-identical via FLR EXPORT (non-destructive), without deleting the originals first.

## Objective
Verify a user can recover specific files/folders from an existing FLB recovery point back to
their original location, restoring content exactly — proving the recovery *mechanism* works
(the "accidentally deleted, then restored" framing is simulated, not literally reproduced, per
the Feasibility note above).

- Repository: **Onboard** (`BACKUP_REPOSITORY-2`)

## Steps
### Step 1 — Build — R4c: `sourceVid=PM-3`, MappingBuilder select `C:/TestData_ForFLB/ft_word`
  (`FOLDER`), `targetStorageVid=BACKUP_REPOSITORY-2`, name `AUTO_FLB_NJM-70015` → `saveJob`.
### Step 2 — Run — R5 (`runType:ALL`).
### Step 3 — Poll — R6 · **Expect:** `lrState:OK`.
### Step 4 — Verify — R7 · FLR `createSession` → `list` confirms the files are present in the
  recovery point → `recover` with `recoveryType:"EXPORT"` to the CIFS share (`win-fs3`,
  `\\10.10.15.3\SharedData`) → SHA-256 the exported files vs `manifests/manifest-windows.sha256`.
  **Do NOT** use `recoveryType:"ORIGINAL"` — blocked by framework safety (overwrites source).
