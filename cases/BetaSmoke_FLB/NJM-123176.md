# NJM-123176 — [FLB v1] FLR from FLB - Functional - Verify Recovery from an Immutable S3 Backup

- **Source TC:** NJM-123176 (Xray, FLB/Share, Blocker P1, flb-v1)
- **Feature:** File-Level Backup (physical machine)
- **Appliance:** nbr-84 (FLB) — see `test-data/environment.md`
- **Metadata:** testcase_id=NJM-123176, author=tri.ton, date=2026-07-08, product=FLB, status=GENERATED
- **Feasibility:** ✅ RUNNABLE (upgraded from BLOCKED, 2026-07-08) — **`Amazon_Immutable`**
  (`BACKUP_REPOSITORY-10`, `objectLockSupported:true`) was added to nbr-84, resolving the repo gap.
  The TC's final step is "recover to original location," which `recoveryType:"ORIGINAL"` +
  Rename overwrite-behavior genuinely satisfies (Rename is a real, non-destructive execution of
  *this exact recovery type*, not a substitute for a different one — confirmed working on a
  Windows physical source, `recipes/file-backup-recipes.md` R7). Overwrite specifically stays
  policy-blocked, but nothing about "recover to original location" itself is reduced in scope by
  using Rename. Open item, as with every other immutability case in this batch: whether NBR
  actually produces an *immutable* savepoint on `Amazon_Immutable` hasn't been live-tested yet.

## Objective
Verify an FLB recovery point stored in an immutable (Object Lock) Amazon S3 repository can be
recovered to its original location, and that the S3 immutability lock remains intact afterward.

## Fixtures used
- Source: windows-src **PM-3** (a dedicated, non-shared file — do not reuse a manifest-tracked
  fixture, to keep this test isolated from other runbooks' checksum oracle)
- Repository: **Amazon_Immutable** (`BACKUP_REPOSITORY-10`) — added 2026-07-08, `objectLockSupported:true`
- Job build: **R4c** + `options.retentionPolicy.retentionMode="RULESET"` (immutable retention —
  introspect via `describe_method` first, this combination is uncalibrated)
- Job name: `AUTO_FLB_NJM-123176`

## Preconditions
- [ ] PM-3 present & OK — **R1**; `Amazon_Immutable` repo online — **R3**.
- [ ] **BLOCKED guard:** if `saveJob` rejects the immutable-retention option shape, or the
  resulting savepoint does NOT show `isImmutableSavepoint:true` / `backupImmutabilitySupport`
  flips true on the repo, stop and report BLOCKED — do not claim immutability was verified from
  the repo's Object Lock capability alone.

## Steps
### Step 1 — Introspect — `mcp__nbr__describe_method nbr-84 JobManagement saveJob` (confirm
  immutable-retention field shape before first use this session).
### Step 2 — Build — R4c: `sourceVid=PM-3`, MappingBuilder select the dedicated test file/folder,
  BackupOptionsBuilder sets `retentionPolicy.retentionMode="RULESET"` (+ immutable count per the
  confirmed shape), `targetStorageVid=BACKUP_REPOSITORY-10`, name `AUTO_FLB_NJM-123176` → `saveJob`.
### Step 3 — Run — R5 (`runType:ALL`).
### Step 4 — Poll — R6 · **Expect:** `lrState:OK`.
### Step 5 — Verify immutability — `JobManagement.getSavepoints` on the new backup object;
  **Expect:** `isImmutableSavepoint:true`. If false, the immutable-retention option didn't actually
  take effect — report FAIL, not a silent pass.
### Step 6 — Recover — FLR `createSession`/`list`, then **`recoveryType:"ORIGINAL"` + Overwrite
  behavior = Rename** (non-destructive substitute for the TC's literal "original location" step —
  see `recipes/file-backup-recipes.md` R7 for the confirmed `<name>-recovered` naming behavior).
  Confirm the recovered item lands at the original path and the S3 Object Lock is still intact
  (repo still shows the savepoint as immutable/non-removable before its retention expires).

## Cleanup
- On PASS — **R9** `remove [<jobId>, false]` (note: an immutable savepoint may resist removal
  until its lock expires — expected, not a failure).
- On FAIL/BLOCKED — leave artifacts + the extra Rename-created file; record job name/id.

## Expected final verdict
PASS if `lrState:OK`, the savepoint shows `isImmutableSavepoint:true`, and the Rename-recovered
copy matches source content. FAIL (not BLOCKED) if the job runs but immutability doesn't actually
engage — repo *existing* is not the same as immutability *working*.
