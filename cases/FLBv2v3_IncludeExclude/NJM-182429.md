## Objective
Verify that a Path exclusion rule scoped to one subfolder skips everything under that path while
backing up the rest of the parent folder.

- Test data: `C:\TestData_ForFLB\IncludeExclude\WithTemp\` — `keep.txt` (top-level) +

  (`enabledSourceItemsExclusion=true`, `sourceItemsExclusion="C:\TestData_ForFLB\IncludeExclude\WithTemp\temp"`)

## Steps
### Step 1 — Build — R4c: `sourceVid=PM-3`, mapping `C:/TestData_ForFLB/IncludeExclude/WithTemp`
  (FOLDER), exclusion rule = exact path to `WithTemp\temp`, `targetStorageVid=BACKUP_REPOSITORY-2`,
  name `AUTO_FLB_NJM-182429` → `saveJob`. **EXECUTED 2026-07-15:** job id 126. **Pattern used:
  bare folder name `temp`** (no path prefix, no wildcard) — the literal path-prefixed pattern from
  the original design (`C:\TestData_ForFLB\...\WithTemp\temp`) was not tried here since the bare
  name proved sufficient and (per the NJM-182426 investigation) full absolute paths risk hanging
  the job. Unlike the *Inclusion* filter (which only matches the top-level folder name but leaves
  it empty of nested content — see NJM-182426's documented FLR-browse defect), *Exclusion* by bare
  folder name worked cleanly: the excluded subfolder disappeared **entirely** from the tree (not
  just emptied), so this TC's verification never needed to browse into a nested folder.

### Step 4 — Verify — R7 · FLR browse `C:\TestData_ForFLB\IncludeExclude\WithTemp\` →
  **CONFIRMED** only `keep.txt` present; `temp/` folder (and `junk.txt` inside) entirely absent.
