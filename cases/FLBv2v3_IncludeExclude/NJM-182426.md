## Objective
Verify that a Path inclusion rule scoped to one subfolder backs up only that subfolder's
contents, skipping sibling subfolders.

## Fixtures used
- Source: `windows-src` **PM-3** (`Window11`)
- Test data: `C:\TestData_ForFLB\IncludeExclude\PathTest\{A,B,C}\` — one marker file per subfolder
  (to be seeded)
- Repository: **Onboard** (`BACKUP_REPOSITORY-2`)
- Job build: **R4c**, source mapping = the `PathTest` parent folder, + Inclusion
  (`enabledSourceItemsInclusion=true`, `sourceItemsInclusion="C:\TestData_ForFLB\IncludeExclude\PathTest\A"`)

## Preconditions
- [ ] PM-3 present & OK — R1
- [ ] Onboard repo online — R3
- [ ] `PathTest` fixture (A/B/C subfolders) seeded on PM-3 — R0

## Steps
### Step 1 — Build — R4c: `sourceVid=PM-3`, mapping `C:/TestData_ForFLB/IncludeExclude/PathTest`
  (FOLDER, the parent), inclusion rule = exact path to subfolder `A`, `targetStorageVid=BACKUP_REPOSITORY-2`,
  name `AUTO_FLB_NJM-182426` → `saveJob`.
  **EXECUTED 2026-07-15, extensive syntax investigation (jobs 116/118–122):**
  - **job 116** — pattern = full absolute Windows path, no wildcard
    (`C:\TestData_ForFLB\IncludeExclude\PathTest\A`). Run **hung**: `crState:RUNNING` with
    `crProgress` frozen at 609 (6.09%) for 3+ minutes, no failure event logged. Stopped manually
    (`JobManagement.stop {"jobIds":[116]}` → `lrState:STOPPED`), job removed.
  - **job 118** — pattern `A\*` (relative, backslash). Ran to `lrState:OK` normally, but FLR
    showed subfolder `A` present at the `PathTest` root (siblings `B`/`C` correctly absent) —
    **yet `A` itself was empty** (`file_a.txt` missing). `consumed:1` (byte count too small for
    real content).
  - **job 119** — pattern `A/*` (relative, forward slash). Identical result to 118 — slash
    direction doesn't matter; same empty-`A`-folder outcome.
  - **job 120** — pattern = full absolute path *with* trailing wildcard
    (`C:/TestData_ForFLB/IncludeExclude/PathTest/A/*`). Ran fine but matched **nothing at all**
    (`consumed:0`, FLR root listing empty — not even folder `A` appeared).
  - **job 121** — pattern = exact relative file path, no wildcard (`A\file_a.txt`). Same
    empty-`A`-folder result as 118/119 (`consumed:1`).
  - **job 122 (diagnostic)** — pattern `*` (broadest possible match, sanity check). `consumed:3`
    bytes — matches exactly 3 × 1-byte fixture files (`file_a.txt`/`file_b.txt`/`file_c.txt`),
    proving the backup engine **did** capture all three files' content this time. FLR root
    listing showed **all three** subfolders `A`, `B`, `C` (expected, since `*` matches
    everything) — but browsing *into* **any** of them (`A`, and by the same mechanism
    presumably `B`/`C`) returned **empty**, despite the byte-count proving real content exists
    inside.
  **Root cause isolated:** this is a **FLR directory-listing defect for nested (second-level+)
  subfolder content when the job was built with `options.enabledSourceItemsInclusion` active**
  — distinct from, but analogous to, the previously-documented RPC-built-Linux FLR-browse defect
  (recipes.md R4 note). Here it reproduces on **Windows** and specifically when an Inclusion
  filter is enabled on a multi-level source tree: the backup engine captures the correct bytes
  (evidenced by `consumed` savepoint size), but `FileLevelRecoveryManagement.list` cannot
  enumerate files one level below the folders it does show. This defeats R7 (FLR-browse)
  verification for this TC's fixture shape regardless of which inclusion-pattern syntax is
  correct — **the verification method itself is unreliable here**, not just the pattern syntax.

## Expected final verdict
**FAIL.** Not because the Inclusion-by-path feature is provably broken, but because **R7
verification (FLR browse) could not be completed** for a subfolder-scoped path pattern on this
build — FLR's directory listing returns empty for nested subfolder content whenever the job's
Inclusion filter is active, even when byte-count evidence shows the engine captured real data.
This blocks a definitive PASS/FAIL call on the actual filtering logic and is itself a
reportable defect (FLR-browse reliability), independent of which exact inclusion-pattern syntax
NBR expects for path-based rules (4 different syntaxes tried, all inconclusive under this
verification method).
